# ArkTS V2 状态管理与数据开发规范

## 1. 核心状态装饰器规范
### 1.1 @ObservedV2 与 @Trace (类属性变化观测)
- **规则说明**：使用 `@ObservedV2` 装饰类，配合 `@Trace` 装饰类内部需要观测的属性，实现深层状态的精准观测。
- **最佳实践**：
  - 仅对需要触发 UI 更新或状态监听的属性使用 `@Trace`。
  - 避免在 `@ObservedV2` 类的构造函数中执行复杂的副作用。

### 1.2 @Computed (计算属性)
- **规则说明**：用于声明计算属性，依赖的其他 `@Trace` 属性变化时自动重新计算。
- **约束**：
  - `@Computed` 装饰的方法必须有返回值。
  - 不要在 `@Computed` 方法内部修改其他状态变量（保持纯函数特性）。

### 1.3 @Monitor 与 @SyncMonitor (状态变量修改监听)
- **@Monitor (异步监听)**：用于监听 `@Trace` 属性或状态变量的变化，在当前事件循环末尾或下一次渲染前异步执行。适合处理耗时操作或不需要立即响应的逻辑。
- **@SyncMonitor (同步监听)**：状态变化时立即同步执行回调。
- **约束**：
  - 避免在 `@SyncMonitor` 中执行耗时操作，以免阻塞 UI 线程。
  - 防止在监听回调中再次修改被监听的属性，导致死循环。

### 1.4 @Type (标记类属性的类型)
- **规则说明**：在状态序列化、反序列化或复杂类型推断场景中，使用 `@Type` 明确指定类属性的类型，确保类型安全。

## 2. 全局与持久化状态存储
### 2.1 AppStorageV2 (应用全局UI状态存储)
- **规则说明**：用于在整个应用生命周期内存储和共享跨组件的 UI 状态。
- **最佳实践**：
  - 尽量按模块或功能划分存储的键值，避免命名冲突。
  - 对于不需要持久化的全局状态，优先使用 AppStorageV2。

### 2.2 PersistenceV2 (持久化存储UI状态)
- **规则说明**：用于将关键 UI 状态持久化到本地磁盘，应用重启后可恢复。
- **约束**：
  - 仅持久化必要的数据（如用户偏好、登录状态等），避免将大量业务数据存入 PersistenceV2，以免影响性能。

## 3. 辅助接口与高级用法
### 3.1 对象观测与代理
- **getTarget**：获取被代理对象的原始目标对象，常用于需要传递非代理对象给底层 C++/第三方库的场景。
- **makeObserved**：将普通对象转换为可观测对象。
- **canBeObserved**：检查一个对象是否可以被转换为可观测对象，在调用 `makeObserved` 前建议进行安全校验。

### 3.2 动态监听管理
- **addMonitor / clearMonitor**：在运行时动态添加或移除状态监听器。
- **规范**：
  - 必须成对使用，在组件销毁或不再需要监听时，务必调用 `clearMonitor` 清理，防止内存泄漏。

### 3.3 状态同步与更新控制
- **applySync**：强制同步执行状态变更及相关的计算属性更新。
- **flushUpdates / flushUIUpdates**：手动触发状态或 UI 的批量更新，通常用于规避异步更新带来的时序问题或在特定性能优化场景下使用。

## 4. 常见错误及解决方案
1. **内存泄漏**：使用了 `addMonitor` 但在生命周期结束时未调用 `clearMonitor`。
   - **解决**：在 `aboutToDisappear` 等销毁生命周期中清理监听。
2. **UI 阻塞**：在 `@SyncMonitor` 中执行了复杂的网络请求或大量计算。
   - **解决**：将耗时操作移至 `@Monitor` 或使用异步 Promise。
3. **循环依赖更新**：在 `@Computed` 或 `@Monitor` 中直接/间接修改了自身依赖的值。
   - **解决**：梳理数据流向，确保状态更新是单向的。
